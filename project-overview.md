# Конверт-трек: система учёта передачи бухгалтерских документов

> Документ описывает замысел проекта, архитектуру, модели данных, бизнес-процессы и этапы реализации. Исходная постановка — в `development.md`.

---

## 1. Задача

На предприятии есть несколько филиалов. Закрывающие бухгалтерские документы (УПД, УКД, акты, счёт-фактуры и т.п.) физически перемещаются между филиалами и клиентами через экспедиторов. Часть документов теряется в пути — и без формализованной передачи невозможно установить, где именно они пропали.

Каждый документ при печати из 1С уже содержит уникальный штрихкод. На этом строится решение.

## 2. Решение

Вводится понятие **конверта** — физического носителя с собственным уникальным ШК и номером. На точке отправки:

1. Оператор сканирует ШК каждого документа, который кладётся в конверт.
2. Программа подтягивает по каждому ШК данные документа из 1С (через OData).
3. Оператор «запечатывает» конверт — система фиксирует его состав и печатает:
  - **опись** (бумажная, вкладывается внутрь, подписывается),
  - **этикетку со ШК конверта** (наклеивается снаружи).
4. На точке получения оператор сканирует ШК конверта и последовательно все документы внутри. Программа визуально отмечает сошедшиеся позиции и сигнализирует о расхождениях.

Результат: на каждом этапе жизни конверта зафиксировано, кто, когда и какие документы передал — и в какой момент образовалась недостача.

### Почему отдельная служба, а не модуль внутри Apache

Apache занят публикацией 1С. Расширять его конфигурацию ради Python-приложения — лишняя связанность. Служба поднимает uvicorn на внутреннем порту (например, `127.0.0.1:8080`), при необходимости Apache проксирует внешний URL на неё через `mod_proxy`. Запуск и перезапуск — через `nssm`.

## 5. Ключевые алгоритмы

### 5.1. Конвертация ШК документа в GUID 1С

```python
import uuid

def barcode_to_guid(barcode: str) -> str:
    return str(uuid.UUID(bytes=int(barcode).to_bytes(16, "big")))
```

Валидация: ШК должен быть строго цифровым и укладываться в 16 байт. На некорректном значении — ошибка в UI «ШК не похож на штрихкод документа 1С».

### 5.2. Получение данных документа из 1С

Запрос OData:

```
GET http://<apache-host>/<base>/odata/standard.odata/Document_<Type>(guid'<GUID>')?$format=json
```

Поскольку тип документа заранее неизвестен, предусматриваем одно из двух:

опрашивать последовательно список известных типов (УПД, УКД, Реализация…), пока не найдём.

Ответ кэшируется в `envelope_documents.raw_1c_payload`, чтобы конверт оставался самодостаточным даже при недоступной 1С.

### 5.3. Генерация номера и ШК конверта

Использовать префикс ТА-NNNNNNNNNNNNNNNN - где NNNNNNNNNNNNNNNN - это уникальный номер конверта. нужно делать полностью рандомную нумерацию. но при этом без дубликатов по бд

## 6. Пользовательские сценарии

### 6.1. ПК-клиент: регистрация конверта

1. Оператор открывает веб-страницу, видит две крупные кнопки: **«Новый конверт»** и **«Проверить конверт»**.
2. Нажимает «Новый конверт» → backend создаёт черновик (`status=draft`), возвращает номер и ШК, UI отображает их сверху.
3. Фокус стоит на скрытом input-поле, ловящем ввод сканера. Оператор сканирует ШК документа → JS собирает строку до `Enter` (терминатор у keyboard-mode сканеров) → отправляет на `/api/envelopes/{id}/documents`.
4. Backend конвертирует в GUID, запрашивает 1С, сохраняет, возвращает карточку документа. UI дописывает строку в таблицу.
5. Повтор для каждого документа. Любую строку можно удалить крестиком (эндпоинт `DELETE /api/envelopes/{id}/documents/{doc_id}`).
6. По кнопке «Запечатать» **или** сканированию ШК самого конверта — backend переводит статус в `sealed`, печатает опись и этикетку. Перед запечатываением также открывается окошко с выбором двух подписантов и места отправки конверта
7. UI показывает подтверждение и возврат к исходному экрану.

**Важно для keyboard-mode сканеров:** страница не должна терять фокус. Любой клик мимо возвращает фокус в скрытое поле. Для отличия «ШК документа» от «ШК конверта» при запечатывании — в буфер-чтении сравниваем префикс/длину либо используем явный режим «ждём ШК конверта».
***Важно***
После запечатывания конверта возможности редактирования состава конверта должен быть заблокирован.

### 6.2. ПК-клиент: проверка конверта

1. Оператор нажимает «Проверить».
2. Сканирует ШК конверта → backend находит конверт по ШК, возвращает его состав. UI отображает список документов серым цветом.
3. Оператор сканирует ШК каждого документа. Успешное сканирование подсвечивает строку зелёным, проставляет `scanned_at_verification`.
4. Если оператор нажимает «Завершить», пока отсканированы не все:
  - Всплывает модалка: «Не отсканировано: N документов. Продолжить сверку / Завершить с расхождением».
  - При завершении с расхождением статус → `verified_with_discrepancy`, в `audit_log` — перечень несошедшихся документов.
5. Если отсканирован документ, которого нет в конверте, — красное предупреждение и запись в лог.

### 6.3. ТСД-клиент

Функционально повторяет ПК-клиент, но интерфейс адаптирован:

- Крупные кнопки, минимум ввода с клавиатуры.
- Сканер работает через DataWedge/Intent — нет проблемы «потерять фокус».
- Печать с ТСД — опциональная, через Bluetooth к мобильному принтеру этикеток. Если принтер недоступен, ТСД позволяет только регистрировать и проверять; печать описи/этикетки делается с ПК.
- Синхронизация: прямая работа с backend по Wi-Fi. Оффлайн-режим в MVP не закладываем, но модель данных позволяет добавить очередь отложенных операций.

## 8. Печатные формы

### Опись конверта (A4)

- Шапка: «Опись конверта № {number}»
- ШК конверта (крупно), дата и время создания
- Филиал-отправитель / получатель
- Таблица документов: № п/п | тип | номер | дата | связанная реализация
- Поля: «Сдал ______ / Принял ______ / Дата ______»
- Подвал: «Печать сформирована {now}»

Реализация: шаблон Jinja2 → HTML → PDF (WeasyPrint). Отправка на принтер — через системное API Windows (`win32print`) либо PDF отдаётся в браузер и печатается пользователем — выберем по удобству на площадке.

### Этикетка ШК (термоэтикетка ~58×40 мм)

- Код в Code128, номер конверта текстом, дата создания.
- Если стоит термопринтер Zebra/TSC — отправляем ZPL/TSPL напрямую по LPT/TCP.
- Если обычный A4 — печатаем лист с несколькими этикетками или одну с крупным кодом.

## 10. Безопасность и устойчивость

- **Аутентификация**: базовая — логин/пароль оператора, сессия в cookie. Для ТСД — тот же API, токен. На старте допустим «мягкий» режим (ввод имени в начале смены), в v2 — нормальная БД пользователей с ролями (`operator`, `admin`).
- **Роль `admin`** — только у неё доступ к `POST /api/admin/reset` и просмотру `audit_log`.
- **Логирование**: каждое событие по конверту — в `audit_log`. Файловый лог службы — через `logging`, ротация средствами Windows.
- **Резервное копирование**: отдельно, `pg_dump` по расписанию Task Scheduler.
- **Отказ 1С**: при недоступности OData добавление документа возможно в «отложенном» режиме — ШК/GUID сохраняем, метаданные подтягиваем позже. В MVP — просто ошибка «1С недоступна, попробуйте позже».

Тонкости печати через тсд - не будем подключать их напрямую к принтерам, вся печать идет через , а тсд только отправляет запрос на печать, дальше уже бэкенд отправляет запрос на печать на принт-сервер
